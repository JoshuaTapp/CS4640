Overview of the Design Process

  1. Design Database Schema
    1. Ideas/data
      1. Must interact with users and domain experts to characterize the data
    1. High-level design
      1. Translate requirements into Conceptual Model
        1. Typically use: E-R Diagrams
        1. this process also produces functional requirements
          1. The requirements of modifying the data in the database (add, delete, update)
      1. Convert model into Relational Model (schema and constraints)
    1. Database Model
      1. Normalize and develop conceptual (logical) schema
        1. Logical-Design Phase
        1. map conceptual schema into relation schema
      1. Develop Physical Schema (partitioning and indexing)
        1. Physical-Design Phase
        1. Physical features of database
          1. Form of file organization (how to partition files)
          1. Choice of Index structure (how to index those files)

  1. Design program that access and update data
    1. How to ensure data access/update doesnt corrupt database
    1. Cover Later
  1. Design Security schema to control access of data
    1. How to ensure only permissioned access is allowed to data.
    1. Cover Later

E-R Diagram

  • Cheat Sheet
  • Entity: an object, a row in a table
  • Entity Set: a class, a table, a collection of similar entities
  • Attribute: a field, property of the entities in that set
    • Single Value: allows single value
    • Multi-Valued: allows multiple values at the same time
    • Derived: can be calculated from one or more attributes
    • Composite: consists of multiple values

Relationships: connection among 2+ entity sets

  • Binary: connections between 2 entity sets
  • Multi-way (u-ary): connections between more than 2 entity sets.
  • Converting Multi-way to Binary
    • Replace a multiway-way relationship with an entity set and binary relationships
    • Has to respect the reality the database is trying to represent though
  • Cardinality:
    • One: guarantee at most ONE entity in the entity set.
    • Many: guarantee at least ONE entity in the entity set.
  • Participation
    • Total Participation: all entities in an entity set must participate in the relationship
  • Attributes on Relationships
  • Roles in Relationships
    • An entity set can appear two or more times in a single relationship
    • Each edge to the entity set represents a different role that the entity set plays in the relationship
  • Weak Entity
    • A weak entity set is that which cannot be identified by
    • a primary key but must be identified by the combination of its attributes and its relationship to another entity set
    • Example:

      Answer:

      • course(course_id, title, credits)
      • section(course_id, sec_id, semester, year, enroll_limit)

  • Many-to-Many Relationships
    • When you have many-to-many relationships between entity sets, you must create a table to represent the relationship.
    • Example:
      • instructor(instr_id, name, salary)
      • student(S_ID, name, total_cred)
      • advisor(instr_ID, S_ID, date)
  • Many-to-One / One-to-Many:
    • Requires separate table with the primary key of the Many relationship and the attributes of the relationship. (this includes the primary key of the One relationship as an attribute)
    • Example:
      • instructor(instr_id, name, salary)
      • department(dept_id, name)
      • works_for(instr_id, dept_id)

  • Total Participation Relationship Set
    • add the primary key of the one side to the many side entity set. No table for relationship needed
    • Example:
      • product(pid, name, description, cname, quantity)
        • cname and quantity are attributes of company
      • company(cname, address)
    • Example:
      • department(dept_id, name)
      • instructor(instr_id, name, salary, dept_id)
